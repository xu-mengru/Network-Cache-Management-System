### 问题 1：第二组通信协议从 TLV 变更为 RESP（含衍生问题）

| 属性 | 内容 |
|------|------|
| **发现阶段** | Day6 |
| **严重程度** | 严重 |
| **涉及模块** | `net/` 全部（`CacheServerClient`, `TcpCacheClient`, `MockCacheClient`）、`MainController`、`CacheEntry` |
| **提出人** | 第二组组长 |

#### 问题描述

第二组初始设计文档约定使用 **TLV 二进制协议**（Length 4B + Type 1B + ArgCnt 1B + 变长 Args），我们的 `TcpCacheClient` 骨架也基于此格式设计。Day6 第二组发送最终版设计文档（v5.0，2026-07-05），协议**改为 RESP 文本协议**（Redis Serialization Protocol），且支持的命令集从约 14 个缩减为 8 个。

关键差异：
- **编码方式**：二进制 TLV → 文本 RESP（`*n\r\n$len\r\ndata\r\n...`）
- **调试方式**：需专用工具 → **telnet/redis-cli 可直接测试**
- **支持命令**：原有 `EXISTS/EXPIRE/TTL/TYPE/KEYS/GETALL/CLEAR/STATS` 等均不再支持
- **新增命令**：`PING/LPUSH/RPUSH/LPOP/LRANGE` 等 List 操作

#### 分析过程

1. **读取文档**：读取第二组的 `RESP协议文档_8个命令格式说明.md` 和 `网络缓存系统服务端_完整设计文档.md`
2. **逐条对比**：将我们 `CacheServerClient` 接口的 11 个方法逐一与第二组支持的 8 个命令对比
3. **影响评估**：
   - `TcpCacheClient`：TLV 编解码全部废弃 → **100% 重写**
   - `MockCacheClient`：需删除 7 个方法、新增 5 个方法 → **~70% 重构**
   - `MainController`：依赖 `getAll()/stats()/keys()` 的 UI 需重写 → **~40% 重构**
   - `CacheEntry`：需新增 `type` 和 `listLength` 字段

#### 解决方式

1. **协议编解码层重构**：
   - 创建 `protocol/` 子包，新增 `RespCodec.java`（RESP 编解码器）和 `RespResponse.java`（响应类型封装）
   - 废弃 `TcpCacheClient.java`，创建 `RespCacheClient.java`
   - RESP 编码为纯字符串拼接，无需第三方依赖

2. **接口层重构**：
   - 删除 `exists/expire/ttl/type/keys/getAll/clear/stats`
   - 新增 `ping/del(变参)/lpush/rpush/lpop/lrange/ttl`
   - `scan` 方法保留占位（格式待第二组确认）

3. **数据模型扩展**：
   - `CacheEntry` 新增 `EntryType type` 枚举和 `int listLength` 字段

4. **Mock 端重写**：
   - 支持双存储结构（`stringStore` + `listStore`）
   - 新增 `getAllLocalEntries()` 供 UI 本地渲染（不在接口上）

5. **启动器适配**：
   - `CacheClientApp` 中新增 `client.mock` 配置驱动的双实现选择

#### 经验教训

- **接口隔离至关重要**：我们的 `CacheServerClient` 接口层设计使得协议变更的影响隔离在 `net/` 包内，UI 层几乎不需要修改逻辑，只需增删菜单按钮
- **Mock 先行开发策略正确**：即使协议完全变更，Mock 模式下 UI 开发不受影响
- **提前确认协议**：应更早向第二组确认最终协议格式，避免基于未确认的 TLV 协议开发 TcpCacheClient 骨架

### 问题 2：第二组仅同意新增 SCAN 和 TTL 两个命令

| 属性 | 内容 |
|------|------|
| **发现阶段** | Day6 |
| **严重程度** | 中等 |
| **涉及模块** | `CacheServerClient` 接口、`RespCacheClient`、`MainController` |

#### 问题描述

在第二组确认 RESP 协议后，我们提出补充 4 个缺失命令：`EXISTS`、`SCAN`（替代 KEYS）、`EXPIRE`、`TTL`。Day6 第二组组长询问这些是"演示需要还是功能需要"，并最终**仅同意增加 SCAN 和 TTL**。

#### 分析过程

1. **整理缺失命令清单**：
   - `EXISTS`：可用 `GET` 判 null 替代——→ **演示需要**
   - `SCAN`（替代 KEYS）：无替代方案，没有它客户端无法获取 key 列表——→ **功能 + 演示需要**
   - `EXPIRE`：可用 `SET...EX` 替代——→ **演示需要**
   - `TTL`：客户端可本地计时兜底——→ **功能需要（弱）**

2. **谈判策略**：
   - 主动放弃 `EXISTS` 和 `EXPIRE`（降低对方开发压力）
   - 死守 `SCAN`（没有它客户端"废了"）
   - `TTL` 作为适度争取项
   - 提出最低保障方案：如果 `SCAN` 也做不了，至少提供 `DBSIZE` 接口

3. **结果**：第二组同意增加 `SCAN` 和 `TTL`，格式待确认。

4. **后续（Day7）**：第二组发送 `group1_info/TTL_SCAN.md`，最终确认格式：
   - **SCAN**：一次性返回全部匹配 key（非游标式），支持 `*` 通配符
   - **TTL**：`:n`(剩余秒)/`:-1`(持久key)/`:-2`(不存在)，与标准 Redis 完全一致

#### 解决方式

1. `CacheServerClient` 接口中保留 `scan()` 和 `ttl()` 方法签名
2. `MockCacheClient` 中实现完整的 `scan()` 和 `ttl()` 逻辑
3. `RespCacheClient` 中根据确认格式实现 `ttl()`；`scan()` 标记为 `TODO`（归组员B实现）
4. `MainController` 中新增 TTL 查询面板和 PING 按钮
5. `onSearch()` 改为本地过滤（不依赖服务端 KEYS/SCAN）
6. **`docs/网络缓存管理系统_概要设计说明书_正确格式最终版.docx`** 中"待确认接口"表格已更新为"已确认"

#### 经验教训

- **分清演示需要和功能需要**是跟第二组谈判的关键话术
- **主动降低对方开发压力**（放弃 EXISTS/EXPIRE）换来对方同意 SCAN
- **永远准备替代方案**：即使 SCAN 也没有，客户端还可以用本地 key 列表方案

### 问题 3：MainView.fxml 引用已删除的方法导致运行时崩溃

| 属性 | 内容 |
|------|------|
| **发现阶段** | Day6（运行时） |
| **严重程度** | 阻塞 |
| **涉及模块** | `ui/MainController.java`, `MainView.fxml` |

#### 问题描述

重构 `MainController.java` 后，双击 Maven `javafx:run` 启动报错：

```
javafx.fxml.LoadException: 
Error resolving onAction='#onShowStats', 
either the event handler is not in the Namespace or there is an error in the script.
/E:/.../MainView.fxml:25
```

根本原因：重构时删除了 `onShowStats()` 方法及统计面板的 4 个 `@FXML Label` 字段，但 FXML 文件仍然引用这些元素。

此外 FXML 还有其他不匹配：
- `xmlns="http://javafx.com/javafx/26"` 与运行时 JavaFX 21 版本不匹配（产生警告）
- 缺少新增的 `typeColumn`、List 操作面板、TTL 查询面板、PING 按钮的 FXML 定义

#### 分析过程

1. **读取错误堆栈**：定位到 `MainView.fxml:25` 的 `onAction='#onShowStats'`
2. **对比 FXML 和 Controller**：逐行核对 `fx:id` 和 `onAction` 是否一一对应
3. **发现 3 类不匹配**：
   - 删除的方法被引用：`onShowStats`
   - 删除的字段被引用：`statEntriesLabel` 等 4 个
   - 新增的字段和方法缺少 FXML 定义：`typeColumn`、`listKeyField` 等

#### 解决方式

1. **重写 FXML**，重新规划布局：
   - 删除统计面板（`Stats` 按钮 + 4 个统计标签）
   - 新增 PING 按钮 + 结果显示
   - 新增 List 操作行（`listKeyField` / `listValueField` / LPUSH/RPUSH/LPOP/LRANGE）
   - 新增 TTL 查询行（`ttlKeyField` + Query TTL 按钮 + `ttlResultLabel`）
   - 表格新增 `typeColumn` 列
   - 新增 List 结果展示区（`listResultView` ListView）
   - 修正 `javafx/26` → `javafx/21`

2. **验证方式**：逐条核对 FXML-to-Controller 绑定表

#### 经验教训

- **FXML 和 Controller 是双向绑定**，任何一方修改时另一方必须同步更新
- 建议后续在 `MainController` 中添加启动时断言，检查所有 `@FXML` 字段不为 null
- FXML 文件应纳入版本控制，与 Java 代码同步修改

### 问题 4：FXML 文件无法被 Scene Builder 打开

| 属性 | 内容              |
|------|-----------------|
| **发现阶段** | 第5天             |
| **严重程度** | 中等              |

### 问题 5：JavaFX 版本警告（FXML version 26 vs runtime 21）

| 属性 | 内容              |
|------|-----------------|
| **发现阶段** | Day5（首次启动）      |
| **严重程度** | 轻微（不影响运行）       |
| **涉及模块** | `MainView.fxml` |

#### 问题描述

运行时控制台输出警告：

```
Loading FXML document with JavaFX API of version 26 
by JavaFX runtime of version 21.0.6
```

原因：FXML 头部的 `xmlns="http://javafx.com/javafx/26"` 使用了 Scene Builder 最新版本（26），但项目依赖的 JavaFX 运行时是 21。

#### 分析过程

1. 检查 FXML 头部版本声明
2. 检查 `pom.xml` 中 JavaFX 依赖版本（21.0.6）
3. 确认是 Scene Builder 自动写入的版本号

#### 解决方式

将 FXML 头部从：
```xml
<VBox xmlns="http://javafx.com/javafx/26" ...>
```
改为：
```xml
<VBox xmlns="http://javafx.com/javafx/21" ...>
```

#### 经验教训

- IntelliJ IDEA 的 Scene Builder 集成可能使用与项目不同的 JavaFX 版本
- 手动编辑 FXML 时注意版本号与 `pom.xml` 中声明的一致

### 问题 6：配置文件缺少 client.mock 模式切换开关

| 属性 | 内容 |
|------|------|
| **发现阶段** | Day6（接口重构时） |
| **严重程度** | 中等 |
| **涉及模块** | `util/Config.java`, `config.properties`, `CacheClientApp.java` |

#### 问题描述

原设计中 `MainController` 直接硬编码 `MockCacheClient`：

```java
private final CacheServerClient client = new MockCacheClient();
```

这种方式下，切换到 `RespCacheClient` 必须修改源代码，不符合"配置驱动"的设计原则。

#### 分析过程

1. 检查 `Config.java` 和 `config.properties`
2. 发现没有关于"客户端模式选择"的配置项
3. `CacheClientApp` 启动后不创建客户端实例，由 `MainController` 自行创建

#### 解决方式

1. **`config.properties`** 新增：`client.mock=true`
2. **`Config.java`** 新增：`isClientMock()` 方法
3. **`CacheClientApp.java`** 新增全局 `CLIENT` 静态变量，启动时根据配置选择实现
4. **`MainController.java`** 改为从 `CacheClientApp.getClient()` 获取客户端实例

#### 经验教训

- 硬编码是架构灵活性的敌人，即使对教学项目也是如此
- 所有环境相关的选择（Mock / 真实）都应该配置化

### 问题 7：全局静态访问模式不适配未来多客户端架构

| 属性 | 内容 |
|------|------|
| **发现阶段** | Day7（架构审查） |
| **严重程度** | 中等 |
| **涉及模块** | `CacheClientApp.java`, `MainController.java` |

#### 问题描述

原设计中 `CacheClientApp` 持有全局唯一的 `CLIENT` 静态实例，`MainController` 通过 `CacheClientApp.getClient()` 直接访问。这种模式下：

- 只能存在一个客户端实例和一个 Socket 连接
- 所有 UI 组件共享同一个 client，无法为不同标签页提供独立连接
- 组员B 的多客户端标签页任务无法基于此架构实现

具体冲突：

| 维度 | 当前设计 | 多客户端需求 |
|:----:|---------|-------------|
| 实例数 | 全局 1 个 CLIENT | 每个标签页 1 个独立客户端 |
| Socket 数 | 1 个 | 每个标签页 1 个独立 Socket |
| 连接目标 | 单一 host:port | 各标签页可连接不同服务器 |
| UI 获取方式 | `CacheClientApp.getClient()` 硬编码 | 需要按标签页索引获取 |

#### 分析过程

1. 审查 `CacheClientApp.java`：发现 `private static CacheServerClient CLIENT` 是单实例
2. 审查 `MainController.java`：发现 `client = CacheClientApp.getClient()` 是静态硬编码
3. 追溯原始设计：`client.mock` 配置驱动模式（问题 6）解决了 Mock/RESP 切换的问题，但没有解决多实例的问题
4. 评估影响范围：
   - 协议层（RespCodec、RespResponse）：不受影响
   - 接口层（CacheServerClient）：不受影响
   - 实现层（MockCacheClient、RespCacheClient）：每个实例独立持有状态，不受影响
   - **胶水层（CacheClientApp + MainController）**：需要重构
   - 多客户端标签页（组员B 待开发）：当前架构不支持，阻塞开发

#### 解决方式

1. **`CacheClientApp.java`** — 从单实例改为注册表模式：
   - `static CLIENT` → `Map<String, CacheServerClient> CLIENTS`（ConcurrentHashMap）
   - 新增 `createClient(tabId)` — 创建新实例并注册
   - 新增 `getClient(tabId)` — 按标签页 ID 获取独立实例
   - 新增 `removeClient(tabId)` — 关闭标签页时释放 Socket
   - 保留 `getDefaultClient()` — 向后兼容单客户端模式

2. **`MainController.java`** — 从静态访问改为 setter 注入：
   - `final client = getClient()` → 字段声明 + `setClient(tabId, client)` 方法
   - `initialize()` 中增加向后兼容兜底：未注入时自动获取默认客户端
   - 11 个核心文件（协议层、接口层、实现层、工具类）零改动

3. **组员B 待实现**（代码中已留注释）：
   - 创建 `TabPaneController.java`，管理标签页增删切换
   - 新建标签页时调用 `createClient(tabId)` + `setClient(tabId, client)`
   - 关闭标签页时调用 `removeClient(tabId)` 释放资源

#### 经验教训

- **全局静态访问（Global Accessor）不是真正的单例**，它在单客户端场景下可用，但无法平滑扩展为多实例
- **控制器实例化方式**影响架构设计：JavaFX 的 FXMLLoader 通过反射创建 Controller（无参构造），因此依赖注入必须通过 setter 而非构造函数
- **接口隔离 + 注册表模式** 的组合可以无侵入地支持单实例到多实例的扩展，核心业务代码不需要感知这种变化
