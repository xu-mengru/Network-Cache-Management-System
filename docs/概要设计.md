## 1. 引言

本文档旨在描述网络缓存管理客户端系统的整体架构设计、模块划分、技术选型及模块间的交互关系，为后续详细设计与开发提供指导。

### 1.2 适用范围

本文档适用于第二组全体开发人员，以及需要与本客户端联调的第一组（Linux 缓存服务端）开发人员。

### 1.3 术语表

| 术语 | 说明 |
|------|------|
| Cache Entry | 缓存条目，包含 key、value、TTL、创建时间 |

| Mock | 模拟实现，用于界面先行开发 |
| TCP 协议 | 与第一组服务端通信的自定义应用层协议 |

## 2. 总体架构

### 2.1 系统定位

本系统为**实习题目 2 —— 网络缓存管理客户端**，通过自定义 TCP 协议与**题目 1（Linux 缓存服务端）**通信，为用户提供图形化的缓存数据管理能力。

### 2.2 架构风格

采用**分层架构**，自顶向下分为三层：

```
┌─────────────────────────────────────────────┐
│              界面层 (UI Layer)               │
│    JavaFX + FXML + Scene Builder             │
│    MainView.fxml / MainController            │
├─────────────────────────────────────────────┤
│              接口层 (Interface Layer)         │
│    CacheServerClient (接口契约)              │
├─────────────────────────────────────────────┤
│              实现层 (Implementation Layer)    │
│    ┌─────────────────┐ ┌─────────────────┐  │
│    │  MockCacheClient │ │  TcpCacheClient  │  │
│    │  (开发调试用)     │ │  (联调/生产用)   │  │
│    └─────────────────┘ └─────────────────┘  │
└─────────────────────────────────────────────┘
```

### 2.3 设计原则

- **接口隔离**：界面层仅依赖 `CacheServerClient` 接口，不依赖具体实现
- **可替换性**：通过 `MockCacheClient` / `TcpCacheClient` 切换，支持界面先行开发
- **单一职责**：每个模块职责清晰，界面不涉及网络通信逻辑

## 3. 模块划分

### 3.1 模块列表

| 模块 | 包路径 | 职责 | 优先级 |
|------|--------|------|--------|
| 数据模型 | `model` | CacheEntry 实体定义 | P0 |
| 服务接口 | `net` | CacheServerClient 接口契约 | P0 |
| Mock 实现 | `net` | 本地模拟缓存行为 | P0 |
| TCP 实现 | `net` | 与 Linux 服务端的网络通信 | P1 |
| UI 控制器 | `ui` | JavaFX 界面事件处理 | P0 |
| FXML 视图 | `resources/...` | 界面布局定义 | P0 |
| 工具类 | `util` | 序列化、配置读取等 | P1 |
| 测试 | `test/...` | 单元测试 / 集成测试 | P0 |

### 3.2 模块依赖关系

```
ui (Controller)
  ↓ 依赖
net (CacheServerClient 接口)
  ↓ 实现
net/MockCacheClient      net/TcpCacheClient
  ↓ 依赖                      ↓ 依赖
model (CacheEntry)        model (CacheEntry)
                              ↓ 依赖
                          util (协议编解码)
```

## 6. 通信协议草案

> **说明**：以下为客户端提出协议草案，待与第一组协商确认后正式定稿。

### 6.1 传输层

- TCP，客户端主动发起连接
- 默认端口：**6379**（参考 Redis 端口，最终由第一组确定）

### 6.2 消息格式（建议）

请求格式：
```
<操作> <参数1> <参数2> ...\r\n
```

响应格式：
```
<状态码> <数据>\r\n
```

### 6.3 基本操作映射

| 操作 | 请求示例 | 成功响应示例 |
|------|---------|-------------|
| GET | `GET key1\r\n` | `+OK value\r\n` |
| SET | `SET key1 value1 300\r\n` | `+OK\r\n` |
| DEL | `DEL key1\r\n` | `+OK 1\r\n` |
| KEYS | `KEYS user:*\r\n` | `+OK key1,key2,key3\r\n` |
| STATS | `STATS\r\n` | `+OK entries:128\r\n` |
| CLEAR | `CLEAR\r\n` | `+OK\r\n` |

> 具体协议格式以第一组最终确认为准，此处仅用于驱动客户端界面开发。

## 7. 非功能需求

### 7.1 性能

- 界面操作响应 < 100ms（Mock 模式下）
- 支持管理 10 万级别缓存条目

### 7.2 可用性

- 连接中断时显示明确状态提示
- 支持自动重连（可配置）

### 7.3 可维护性

- `CacheServerClient` 接口文档完整
- TCP 实现层独立可替换
