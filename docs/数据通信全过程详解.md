## 二、外部通信：客户端 ↔ 第一组服务端

### 2.1 传输层

| 属性 | 值 | 原因 |
|------|-----|------|
| 协议 | TCP | RESP 基于 TCP，可靠、有序、面向连接 |
| 默认地址 | 127.0.0.1:6379 | 教学环境局域网；6379 是 Redis 惯例端口 |
| 连接方式 | 客户端主动发起 connect() | 服务端监听，客户端连接（C/S 模型） |
| 超时 | 5000ms（config.properties 配置） | 避免用户无限制等待 |

### 2.2 应用层协议：RESP

**为什么选 RESP 而不是自定义 TLV？**（第一组的决策）

| 维度 | RESP（文本） | TLV（二进制） |
|------|:-----------:|:------------:|
| 解析代码量 | ~50 行 | ~150 行 |
| 调试工具 | telnet 直接敲 | 需要写专用解码器 |
| redis-cli 兼容 | ✅ 完全兼容 | ❌ |
| 带宽效率 | 中（文本冗余） | 高 | 
| 9 天开发周期内 | ✅ 可快速交付 | ❌ 开发时间不够 |

### 2.3 请求格式（客户端 → 服务端）

```
通用格式：
  *<参数个数>\r\n
  $<参数1长度>\r\n
  <参数1>\r\n
  $<参数2长度>\r\n
  <参数2>\r\n
  ...

实际示例：SET name Alice（从网线上抓到的字节）
  2a 33 0d 0a 24 33 0d 0a 53 45 54 0d 0a 24 34 0d 0a
  *  3  \r \n  $  3  \r \n  S  E  T  \r \n  $  4  \r \n
  6e 61 6d 65 0d 0a 24 35 0d 0a 41 6c 69 63 65 0d 0a
  n  a  m  e  \r \n  $  5  \r \n  A  l  i  c  e  \r \n
```

### 2.4 响应格式（服务端 → 客户端）

| 首字节 | 名称 | 格式 | 对应 Java 类型 | 何时出现 |
|:------:|------|------|:-------------:|---------|
| `+` | 简单字符串 | `+OK\r\n` | String | SET 成功、PING |
| `-` | 错误 | `-ERR msg\r\n` | RuntimeException | 命令不存在、参数错误 |
| `:` | 整数 | `:3\r\n` | long | DEL 返回数量、LPUSH 返回长度 |
| `$` | 批量字符串 | `$5\r\nhello\r\n` | String 或 null | GET 返回值 |
| `$` | 空值 | `$-1\r\n` | null（Java 端） | GET 不存在的 key |
| `*` | 数组 | `*2\r\n...` | List\<String\> | LRANGE 返回值 |

### 2.5 12 个操作的完整网络流量

以下每行代表一次完整的 TCP 往返（客户端发送 → 服务端处理 → 服务端响应）：

```
① PING
   发: *1\r\n$4\r\nPING\r\n
   收: +PONG\r\n
   耗时: ~1ms（局域网）
   说明: 最简单的请求，不涉及数据库操作

② GET key
   发: *2\r\n$3\r\nGET\r\n$3\r\nkey\r\n
   收: +OK\r\n               ← 服务端返回实际 value
   收: $-1\r\n              ← key 不存在
   说明: 查 String 类型时发一次，收一次

③ SET key value [EX sec]
   发: *3\r\n$3\r\nSET\r\n$3\r\nkey\r\n$5\r\nvalue\r\n
   收: +OK\r\n
   说明: 写入操作，服务端返回 OK 表示确认

④ DEL key1 key2
   发: *3\r\n$3\r\nDEL\r\n$4\r\nkey1\r\n$4\r\nkey2\r\n
   收: :2\r\n               ← 实际删除了 2 个
   说明: 变参命令，一次网络请求删除多个

⑤ TTL key
   发: *2\r\n$3\r\nTTL\r\n$3\r\nkey\r\n
   收: :60\r\n              ← 剩余 60 秒
   收: :-1\r\n              ← 持久 key
   收: :-2\r\n              ← key 不存在

⑥ LPUSH key a b c
   发: *4\r\n$5\r\nLPUSH\r\n...\r\n
   收: :3\r\n               ← 插入后的列表长度

⑦ LRANGE key 0 -1
   发: *4\r\n$6\r\nLRANGE\r\n...\r\n$1\r\n0\r\n$2\r\n-1\r\n
   收: *3\r\n$1\r\na\r\n$1\r\nb\r\n$1\r\nc\r\n
   说明: 数组响应，逐个解析每个元素
```

```

  数据: keyField 文本 = "name"
        valueField 文本 = "Alice"
        ttlField 文本 = "300"
  流向: JavaFX 双向绑定，字段以 String 形式存在 TextField 中

步骤 1：事件触发
  组件: MainView.fxml > Button(onAction="#onAdd")

    String value = valueField.getText().trim()   → "Alice"
    long ttl = Long.parseLong("300")             → 300
  校验: key 和 value 非空才继续（防止无效请求）

步骤 2：接口调用（多态分发）
  组件: CacheServerClient 接口（代码中的引用）
  方法: client.set("name", "Alice", 300)
  这里的 client 的实际类型由 CacheClientApp 启动时决定：
    ① Mock 模式 → client = MockCacheClient 实例
    ② RESP 模式 → client = RespCacheClient 实例

步骤 3A（Mock 分支）：内存写入
  组件: MockCacheClient
  方法: set(key, value, ttlSeconds)

    listStore.remove(key);
    // 创建新条目写入 stringStore
    CacheEntry entry = new CacheEntry("name", "Alice", 300);
    stringStore.put("name", entry);
  存储状态变化:
    stringStore: {..., "name" → CacheEntry("name", "Alice", 300)}
  返回: true

步骤 3B（RESP 分支）：网络写入
  组件: RespCacheClient
  方法: set("name", "Alice", 300)
  内部调用:

    execute("SET", "name", "Alice", "EX", "300")
  
  execute() 内部展开:
    // 3B-1：RESP 编码
    byte[] request = RespCodec.encode("SET", "name", "Alice", "EX", "300")
      → 字符串拼接: "*5\r\n$3\r\nSET\r\n$4\r\nname\r\n$5\r\nAlice\r\n$2\r\nEX\r\n$3\r\n300\r\n"
      → 编码为 UTF-8 字节数组

    // 3B-2：TCP 发送
    out.write(request);   → 写入 Socket 输出流
    out.flush();          → 强制发送到网络

    // 此时数据已离开 Java 进程，进入 TCP 协议栈
    // → 网卡 → 交换机 → 第一组的 Linux 机器 → 网卡 → 服务端程序

      → 读取首字节: '+' → 知道是简单字符串
      → 读取到 \r\n: "OK"
      → 构造 RespResponse(type=SIMPLE_STRING, value="OK")

    // 3B-4：响应判断
    resp.isError()? → false（不是 -ERR）
    return true

  网络状态变化:
    服务端内存: dictAdd("name", "Alice") + expire set 300s

步骤 4：UI 刷新
  组件: MainController
  方法: refreshTable() (line 245-248)
  数据流转:

      → Mock 模式: mock.getAllLocalEntries()
        → 遍历 stringStore + listStore → 组装 List<CacheEntry>
      → RESP 模式: SCAN + 逐个 GET（待 SCAN 格式确认后实现）
    
    tableData.setAll(all);     → 更新 ObservableList
    tableView.setItems(tableData);  → JavaFX 自动重新渲染 TableView
```

## 四、关键组件中的数据正确性保障

### 4.1 RespCodec.encode() — 请求编码正确性

```java
public static byte[] encode(String... args) {
    StringBuilder sb = new StringBuilder();
    sb.append('*').append(args.length).append("\r\n");
    for (String arg : args) {
        byte[] bytes = arg.getBytes(StandardCharsets.UTF_8);
        sb.append('$').append(bytes.length).append("\r\n");
        sb.append(arg).append("\r\n");
    }
    return sb.toString().getBytes(StandardCharsets.UTF_8);
}
```

**保障机制：**

| 机制 | 说明 |
|------|------|
| **参数个数精确** | `args.length` 决定了 `*n` 中的 n，与服务端期望的 argc 一致 |
| **长度精确** | `arg.getBytes(UTF-8).length` 计算的是 UTF-8 编码后的真实字节长度，不是字符长度（中文字符 = 3 字节） |
| **分隔符严格** | `\r\n` 是 RESP 协议强制要求的行分隔符 |
| **UTF-8 一致性** | 编码时指定 UTF-8，保证中文/特殊字符不丢失 |

### 4.2 RespCodec.decode() — 响应解码正确性

```java
public static RespResponse decode(InputStream in) throws IOException {
    int firstByte = in.read();     // ① 首字节决定响应类型
    return switch (firstByte) {
        case '+' -> decodeSimpleString(reader);   // 读一行
        case '-' -> decodeError(reader);          // 读一行，标记为错误
        case ':' -> decodeInteger(reader);        // 读一行，转 long
        case '$' -> decodeBulkString(reader);     // 读长度 + 读取N字节 + 跳过\r\n
        case '*' -> decodeArray(in, reader);      // 递归解析每个元素
        default -> throw new IOException("Unknown type");
    };
}
```

**保障机制：**

| 情况 | 风险 | 保护措施 |
|------|------|---------|
| 半包（一次 recv 没收到完整响应） | 读取不完整，解析错位 | `BufferedReader` 按行读取，天然处理半包；阻塞直到完整行到达 |
| 粘包（一次 recv 包含多条响应） | 读取超出本条 | 批量字符串按 `$len` 精确读取 len 字节，不读多；数组按 `*n` 精确解析 n 个元素 |
| 空值 $-1 | 误判为普通字符串 | 显式检查 `length == -1` → 返回 `RespResponse.nullBulk()` |
| 连接断开 | 读到 -1 | 检测 `firstByte == -1` → 抛出 `IOException("Connection closed")` |

### 4.3 RespCacheClient.execute() — 收发同步正确性

```java
private RespResponse execute(String... args) {
    byte[] request = RespCodec.encode(args);  // ① 编码
    out.write(request);                        // ② 发送
    out.flush();                               // ③ 刷新
    return RespCodec.decode(in);               // ④ 阻塞等待并解析
}
```

**保障机制：** RESP 是一问一答协议，客户端发完一个请求后必须等响应回来才能发下一个。`execute()` 是**同步阻塞**的：发送后立即阻塞在 `decode()` 上，直到响应到达。这天然避免了请求/响应错位。

**如果不需要 await/async**：教学场景下，每次操作体量极小（微秒级），UI 线程同步阻塞不会产生可感知的卡顿。

### 4.4 MainController.getAllEntries() — 数据一致性

```java
// Mock 模式
private List<CacheEntry> getAllEntries() {
    if (client instanceof MockCacheClient mock) {
        return mock.getAllLocalEntries();  // 直接从内存中读取最新数据
    }
    // TODO: RESP 模式
}
```

**Mock 模式的正确性：** `getAllLocalEntries()` 直接读 `stringStore` 和 `listStore` —— 这是写入数据的原始容器。任何 `set()`/`del()`/`lpush()` 等操作都立即反映到容器中，`getAllLocalEntries()` 调用时拿到的一定是**最新状态**。

**RESP 模式的正确性（待实现）：** 每次刷新重新从服务端 SCAN + GET，天然与服务端状态一致——不存在缓存不一致问题（因为根本不缓存服务端数据，每次都实时拉取）。

### 4.5 TCP 连接状态管理

```java
// RespCacheClient.connect()
socket.connect(new InetSocketAddress(host, port), timeout);
socket.setSoTimeout(timeout);  // 读操作超时

// RespCacheClient.isConnected()
return connected && socket != null && socket.isConnected() && !socket.isClosed();
```

| 状态 | isConnected() 返回 | 后续操作 |
|------|:-----------------:|---------|
| 已 connect 且 socket 正常 | `true` | 正常执行命令 |
| 已 connect，服务端崩溃 | `false`（TCP RST 后检测到） | 按钮变灰，提示"连接断开" |
| 未 connect | `false` | 按钮提示"请先连接" |
| connect 超时 | 抛出异常 | 弹窗"连接超时" |

## 六、"从外部系统获取数据"的完整回答

### 所有外部数据获取途径汇总

| 获取方式 | 涉及操作 | 网络请求数 | 说明 |
|---------|---------|:---------:|------|
| **单步查询** | `GET key` | 1 次 | 查一个已知 key 的值 |
| **单步查询** | `TTL key` | 1 次 | 查一个 key 的剩余时间 |
| **单步查询** | `LRANGE key 0 -1` | 1 次 | 获取整个列表 |
| **单步写入** | `SET key value EX sec` | 1 次 | 写一条数据 |
| **单步写入** | `LPUSH key value` | 1 次 | 列表插入 |
| **批量获取** | `SCAN 0 MATCH *` + 逐个 GET | 1 + N 次 | 获取全部条目（N=key 数）|
| **批量删除** | `DEL key1 key2 key3` | 1 次 | 变参，一次删多个 |
| **连通性** | `PING` | 1 次 | 不操作数据 |

### 当前工作模式对比

| 模式 | 数据获取 | 是否需要网络代码 | 当前状态 |
|:----:|---------|:--------------:|:--------:|
| **Mock** | 从 HashMap 直接读取 | ❌ 不需要 | ✅ 已实现 |
| **RESP** | 通过 TCP Socket 收发 RESP 消息 | ✅ RespCodec + RespCacheClient 已实现 | ✅ 编解码层已实现 |
| **RESP-SCAN** | 通过 SCAN + GET 批量加载 | ⏳ 需要确认 SCAN 格式后补全 getAllEntries() | ❌ 待完成 |

### "需要添加从网络上读取数据的功能吗？"

**不需要。`RespCacheClient` 已经提供了完整的网络读写能力。** 每个方法（get/set/del/lpush/lrange/ttl/ping）都在内部完成了 "编码→发送→等待→接收→解码" 的完整网络流程。你不需要写任何额外的 Socket 代码。

唯一需要补充的代码是：**SCAN 格式确认后，在 `getAllEntries()` 中组合 SCAN + GET 来实现"加载全部数据"**——这是应用层逻辑的组合，不是新的网络功能。
